# API

Документ будет описывать API backend-части проекта **ВкусноТут**.

На текущем этапе API ещё не реализовано. Ниже зафиксирован планируемый набор эндпоинтов.

## Общие принципы

API должно:

- отдавать только опубликованные рецепты для публичной части;
- не отдавать TypeORM Entity напрямую;
- использовать DTO для входящих данных;
- использовать response DTO или мапперы для исходящих данных;
- поддерживать фильтрацию и пагинацию;
- разделять публичные и защищённые эндпоинты.

## Публичные эндпоинты

### Рецепты

Планируемые эндпоинты:

```txt
GET /recipes
GET /recipes/:slug
```

`GET /recipes` должен поддерживать:

- пагинацию;
- поиск;
- фильтр по категории;
- фильтр по тегам;
- фильтр по КБЖУ в будущем.

`GET /recipes/:slug` должен отдавать:

- основную информацию рецепта;
- категорию;
- теги;
- ингредиенты;
- шаги приготовления;
- КБЖУ;
- изображение.

### Категории

Планируемые эндпоинты:

```txt
GET /categories
GET /categories/:slug
```

### Теги

Планируемые эндпоинты:

```txt
GET /tags
```

## Защищённые эндпоинты

Будут доступны после авторизации.

### Избранное

Планируемые эндпоинты:

```txt
GET /favorites
POST /favorites/:recipeId
DELETE /favorites/:recipeId
```

### Мои рецепты

Планируемые эндпоинты:

```txt
GET /my/recipes
POST /my/recipes
PATCH /my/recipes/:id
POST /my/recipes/:id/submit
```

## Административные эндпоинты

Будут доступны ролям:

- `ADMIN`;
- `SUPER_ADMIN`.

Планируемые разделы:

- управление рецептами;
- управление категориями;
- управление тегами;
- управление ингредиентами;
- управление единицами измерения;
- управление пользователями.

## Модерация

Планируемые эндпоинты:

```txt
GET /admin/recipes/pending
POST /admin/recipes/:id/approve
POST /admin/recipes/:id/reject
```

## Swagger/OpenAPI

Для backend подключена Swagger/OpenAPI-документация.

Swagger UI доступен с учётом глобального API-префикса:

```txt
/{apiPrefix}/swagger
```

Если `apiPrefix=api`, итоговый путь:

```txt
/api/swagger
```

OpenAPI-документация используется для описания текущих и будущих HTTP-ручек backend.

## Categories API

### Получить список активных категорий

```txt
GET /{apiPrefix}/categories
```

Если `apiPrefix=api`, итоговый путь:

```txt
GET /api/categories
```

Ручка возвращает список активных категорий рецептов.

Категории отсортированы по:

1. `sortOrder` по возрастанию;
2. `name` по возрастанию.

Пример ответа:

```json
[
  {
    "id": 1,
    "name": "Завтраки",
    "slug": "breakfasts",
    "description": "Рецепты для завтрака.",
    "sortOrder": 10
  }
]
```

Ручка возвращает только активные категории.

Поля `isActive`, `createdAt` и `updatedAt` не входят в публичный ответ текущей ручки, так как относятся к внутренней модели хранения данных.

### Назначение ручки

Ручка нужна для публичной части сайта, чтобы frontend мог получить список разделов каталога рецептов.

В будущем категории будут использоваться:

- в навигации;
- в фильтрах рецептов;
- на страницах категорий;
- при создании и редактировании рецептов.

## Tags API

### Получить список активных тегов

```txt
GET /{apiPrefix}/tags
```

Если `apiPrefix=api`, фактический путь:

```txt
GET /api/tags
```

Ручка возвращает только активные теги.

Теги отсортированы по:

1. `sortOrder` по возрастанию;
2. `name` по возрастанию.

Пример ответа:

```json
[
  {
    "id": 1,
    "name": "ПП",
    "slug": "pp",
    "description": "Рецепты для правильного питания.",
    "sortOrder": 10
  },
  {
    "id": 2,
    "name": "Быстро",
    "slug": "bystro",
    "description": "Рецепты, которые можно приготовить за короткое время.",
    "sortOrder": 20
  }
]
```

Публичный ответ описан через `TagResponseDto`.

В ответ не включаются технические поля:

- `isActive`;
- `createdAt`;
- `updatedAt`.

Swagger-документация доступна по пути:

```txt
/{apiPrefix}/swagger
```

Если `apiPrefix=api`:

```txt
/api/swagger
```

С учётом текущего backend-порта локальный адрес обычно выглядит так:

```txt
http://localhost:5000/api/swagger
```

### Назначение ручки

Ручка нужна для публичной части сайта, чтобы frontend мог получить список дополнительных признаков рецептов.

В будущем теги будут использоваться:

- в фильтрах каталога;
- на странице рецепта;
- при создании и редактировании рецепта;
- в поиске и подборках.

## Units API

### Получить список активных единиц измерения

```txt
GET /{apiPrefix}/units
```

Если `apiPrefix=api`, фактический путь:

```txt
GET /api/units
```

Ручка публичная и используется как справочник для будущих форм рецептов и ингредиентов.

Ручка возвращает только активные записи, у которых:

```txt
isActive = true
```

Сортировка выполняется по:

1. `sortOrder` по возрастанию;
2. `name` по возрастанию.

Пример ответа:

```json
[
  {
    "id": 1,
    "name": "Грамм",
    "shortName": "г",
    "slug": "gram",
    "type": "mass",
    "conversionFactorToBase": 1,
    "sortOrder": 10
  },
  {
    "id": 2,
    "name": "Килограмм",
    "shortName": "кг",
    "slug": "kilogram",
    "type": "mass",
    "conversionFactorToBase": 1000,
    "sortOrder": 20
  }
]
```

Публичный ответ описан через `UnitResponseDto`.

В ответ не включаются технические поля:

- `isActive`;
- `createdAt`;
- `updatedAt`.

Swagger/OpenAPI доступен по адресу:

```txt
http://localhost:5000/api/docs
```
