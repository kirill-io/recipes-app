# Структура базы данных

Документ описывает планируемую структуру базы данных проекта **ВкусноТут**.

База данных: PostgreSQL  
ORM: TypeORM

## Общая идея

Сайт является публичным каталогом рецептов.

Без регистрации пользователь может:

- смотреть опубликованные рецепты;
- искать рецепты;
- фильтровать рецепты по категориям и тегам.

После регистрации пользователь может:

- добавлять рецепты в избранное;
- создавать свои рецепты;
- отправлять рецепты на модерацию.

Администраторы и модераторы могут управлять контентом, ингредиентами, категориями, тегами и публикацией рецептов.

## users

Пользователи сайта.

### Поля

| Поле | Назначение |
|---|---|
| `id` | Уникальный идентификатор |
| `email` | Email пользователя |
| `passwordHash` | Хэш пароля, может быть `null` для OAuth-пользователей |
| `name` | Отображаемое имя |
| `avatarUrl` | Ссылка на аватар |
| `role` | Роль пользователя |
| `status` | Статус пользователя |
| `emailVerifiedAt` | Дата подтверждения email |
| `createdAt` | Дата создания |
| `updatedAt` | Дата обновления |

### Роли

- `USER`
- `TRUSTED_AUTHOR`
- `MODERATOR`
- `ADMIN`
- `SUPER_ADMIN`

### Статусы

- `ACTIVE`
- `BLOCKED`
- `DELETED`

## recipes

Рецепты.

### Поля

| Поле | Назначение |
|---|---|
| `id` | Уникальный идентификатор |
| `authorId` | Автор рецепта |
| `categoryId` | Основная категория |
| `title` | Название рецепта |
| `slug` | ЧПУ-идентификатор для URL |
| `description` | Краткое описание |
| `cookingTimeMinutes` | Время приготовления в минутах |
| `servings` | Количество порций |
| `difficulty` | Сложность |
| `coverImageUrl` | Главное изображение |
| `status` | Статус рецепта |
| `nutritionCalculationMode` | Режим расчёта КБЖУ |
| `cookedWeightGrams` | Вес готового блюда |
| `caloriesPer100g` | Калории на 100 г |
| `proteinsPer100g` | Белки на 100 г |
| `fatsPer100g` | Жиры на 100 г |
| `carbsPer100g` | Углеводы на 100 г |
| `caloriesTotal` | Калории на всё блюдо |
| `proteinsTotal` | Белки на всё блюдо |
| `fatsTotal` | Жиры на всё блюдо |
| `carbsTotal` | Углеводы на всё блюдо |
| `submittedAt` | Дата отправки на модерацию |
| `publishedAt` | Дата публикации |
| `moderatedAt` | Дата модерации |
| `moderatedById` | Кто проверил рецепт |
| `rejectionReason` | Причина отклонения |
| `createdAt` | Дата создания |
| `updatedAt` | Дата обновления |

### Статусы рецепта

- `DRAFT` — черновик;
- `PENDING_REVIEW` — ожидает модерации;
- `PUBLISHED` — опубликован;
- `REJECTED` — отклонён;
- `ARCHIVED` — скрыт/архивирован.

### Режимы расчёта КБЖУ

- `MANUAL` — КБЖУ вводятся вручную;
- `CALCULATED` — КБЖУ рассчитываются по ингредиентам.

## tags

Дополнительные признаки рецептов.

Теги не заменяют категории. Они используются для дополнительной фильтрации и описания рецепта.

Примеры:

- быстро
- много белка
- низкокалорийное
- запекаем
- жарим
- варим
- тушим
- аэрогриль
- блендер
- микроволновка
- без готовки
- для заготовки
- праздничное

### Поля

| Поле | Назначение |
|---|---|
| `id` | Уникальный идентификатор |
| `name` | Название |
| `slug` | ЧПУ-идентификатор |
| `type` | Тип тега |
| `createdAt` | Дата создания |
| `updatedAt` | Дата обновления |

### Возможные типы тегов

- `FEATURE` — особенность рецепта;
- `METHOD` — способ приготовления;

- `SOURCE` — источник или подборка.


Через таблицу:

## ingredient_unit_conversions

Правила перевода единиц измерения в граммы для конкретных ингредиентов.

Эта таблица нужна, чтобы можно было хранить ингредиент в рецепте в удобном виде, например `1 шт.`, но рассчитывать КБЖУ через граммы.

### Примеры

```txt
Яйцо куриное + 1 шт. = 55 г
Лук + 1 шт. = 80 г
Морковь + 1 шт. = 100 г
```

### Поля

| Поле | Назначение |
|---|---|
| `id` | Уникальный идентификатор |
| `ingredientId` | Ингредиент |
| `unitId` | Единица измерения |
| `gramsPerUnit` | Сколько грамм в одной единице |
| `createdAt` | Дата создания |
| `updatedAt` | Дата обновления |

## Текущие таблицы

### `categories`

Создана первая бизнес-таблица проекта — `categories`.

Назначение таблицы: хранение основных категорий рецептов, например завтраки, салаты, супы, ужины, десерты и другие разделы каталога.

Поля таблицы:

- `id` — первичный ключ;
- `name` — название категории;
- `slug` — уникальный URL-идентификатор категории;
- `description` — описание категории;
- `sort_order` — порядок сортировки категории;
- `is_active` — признак активности категории;
- `created_at` — дата создания записи;
- `updated_at` — дата последнего обновления записи.

Для поля `slug` создан уникальный индекс:

```txt
IDX_categories_slug
```

Это нужно, чтобы в системе не могли существовать две категории с одинаковым URL-идентификатором.

Пример будущего использования `slug`:

```txt
/categories/breakfasts
/categories/salads
/categories/desserts
```

Таблица создана через TypeORM-миграцию `CreateCategoriesTable`.

Важно: названия колонок в документации должны соответствовать реальным названиям колонок в миграции. Если в entity для колонок используется `name: 'sort_order'`, `name: 'is_active'`, `CreateDateColumn({ name: 'created_at' })` и `UpdateDateColumn({ name: 'updated_at' })`, то текущая документация соответствует базе.
